<p>Dear QGIS users, enthusiasts and fine people out there. QGIS 3.0 is coming very soon&#8230;.we are in a &#8216;soft freeze&#8217; state at the moment while we wait for some critical last pieces of code to get finalised. Then we go into hard freeze and prepare to roll out our next major release. Those of you that have been playing with the &#8216;2.99&#8217; builds will surely have noticed that QGIS 3.0 is going to feature a huge number of improvements and new features &#8211; both in the user interface and in the API and code internals.</p>

<p style="text-align:center;">But we have a <strong style="font-size:4em;">BIG</strong> problem:<br />
we need your <strong style="font-size:4em;">help</strong> to document and <strong style="font-size:4em;">describe</strong> all those fine new <strong style="font-size:4em;">features!</strong></p>
<p>Yes fine reader now is the time to break out of the &#8216;passive user of QGIS&#8217; mould you might find yourself in and lend a hand. We have an issue tracker with an <a href="https://github.com/qgis/QGIS-Documentation/issues?q=is%3Aopen+is%3Aissue+milestone%3A%22QGIS+3.0%22">issue for each of the new features</a> that has landed in QGIS 3.0. Even if you do not know how to use our Sphinx based documentation system, you can help tremendously by preparing the prose that should be used to describe new features and attaching it to the issue list linked to above. If you do that, the documentation team can do more editorial work and less  &#8216;writing from scratch&#8217; work.</p>
<p>Writing documentation is a brilliant way to enhance your own knowledge of QGIS and learn the new features that are coming in the next release. For those starting out with documentation there are issue reports that are tagged &#8220;<strong>easy</strong>&#8221; to lower the barrier for beginners. If you are an existing documentation team member it would be great if you could review the list and check whether there are more issues that can be tagged as &#8220;easy&#8221;.</p>
<p>The issue list is automatically created whenever a developer commits a change to QGIS with the word &#8216;FEATURE&#8217; in their change notes. In some cases the change may not be something that an end user will be able to see &#8211; so it will be great for volunteers to also review the automatically added issues and <strong>close off any that are not relevant for documentation</strong>.</p>
<p>Other features are quite complex and in some cases could benefit from interaction with the original developer to make sure that the nuances of the new features are properly described. We need documentation writers to follow these thread and present the new functionality in a clear and concise way.</p>
<p>There are some very helpful resources for people just getting started with QGIS documentation. You can read the <a href="http://docs.qgis.org/2.18/en/docs/documentation_guidelines/">documentation for contributors</a>. You can also contact the team via the <a href="https://lists.osgeo.org/mailman/listinfo/qgis-community-team">community mailing list</a> for specific help if the contributor docs don&#8217;t provide the information you need.</p>
<p>If you want to see the <a href="http://docs.qgis.org/2.18/en/docs/user_manual/">QGIS Documentation</a> up-to-date for the version 3.0 release, please do get involved and help Yves Jacolin and the documentation team!</p>
<p>Lastly if you are not able to directly contribute to the documentation, consider <a href="https://www.qgis.org/en/site/getinvolved/governance/sponsorship/sponsorship.html">funding QGIS</a> &#8211; we have a budget for documentation improvements.</p>
<p>We look forward to your support and contributions!</p>
<p>&nbsp;</p>
<p>Tim Sutton (QGIS Project Chairman)</p>
