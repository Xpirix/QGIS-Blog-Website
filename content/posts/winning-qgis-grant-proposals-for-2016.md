<p>We are extremely pleased to announce the winning proposals for our 2016 QGIS.ORG grant programme. Funding for the programme was sourced by you, our p<a href="https://www.qgis.org/en/site/about/sponsorship.html">roject donors and sponsors</a>! <strong>Note:</strong> For more context surrounding our grant programme, please see:</p>
<ul>
<li><a href="http://blog.qgis.org/2016/09/25/update-on-the-qgis-grant-programme/" target="_blank">Update on the QGIS Grant Programme</a></li>
<li><a href="http://blog.qgis.org/2016/08/04/qgis-grants-call-for-applications/">QGIS Grants: Call for applications</a></li>
</ul>
<p>Our intent with the QGIS.ORG Grant Programme is to support work from community that would typically not be funded by client/contractor agreements, and that contributes to the broadest possible swathe of our community by providing cross-cutting, foundational improvements to the QGIS Project.</p>
<p>Voting to select the successful projects was carried out by our QGIS Voting Membership. Each voting member was allowed to select up to 6 of the 18 submitted proposals by means of a ranked selection form. <span style="font-weight:400;">The full list of votes are available </span><a href="https://docs.google.com/a/qgis.org/spreadsheets/d/1chKG-cxe69lPtMqyqXRpLohT0vXGRzXbO-oZge_flXw/edit?usp=sharing"><span style="font-weight:400;">here</span></a><span style="font-weight:400;"> (on the first sheet). The second sheet contains the calculations used to determine the winner (for full transparency). The table below summarizes the voting tallies for those proposals that received one or more votes, along with brief notes on the methodology used:</span></p>

<p>A couple of extra notes about the voting process:</p>
<ul>
<li>Voting was carried out based on the technical merits of the proposals and the competency of the applicants to execute on these proposals.</li>
<li>No restrictions were in place in terms of how many proposals could be submitted per person / organization, or how many proposals could be awarded to each proposing person / organization.</li>
<li>Although the budget for the grant programme was €20,000.00, the total amount for the three winning proposals is €20,500.00 &#8211; an additional €500.00 was made available by the PSC towards the grant programme to accommodate this.</li>
<li>Voting was &#8216;blind&#8217; (voters could not see the existing votes that had been placed).</li>
</ul>
<p>As mentioned in our previous blog post about this selection process, this is the first time that we have asked our newly formed group of QGIS Voting Members to vote. It is extremely gratifying to see such enthusiastic participation in the voting process. Of the 27 voting members, 24 registered their votes. There was one late submission that unfortunately had to be excluded, and 2 non-votes.</p>

<h2><b>Details of the approved grant proposals</b></h2>
<hr />
<h3><b>Implement a flexible properties framework in QGIS (Nyall Dawson) &#8211; €10,000</b></h3>
<p>&nbsp;</p>
<p><b>Details:</b><span style="font-weight:400;"> I am applying for a QGIS grant to cover the implementation of a flexible &#8220;properties framework&#8221; for QGIS. I honestly believe that implementation of this framework will unlock cartographic power in QGIS well beyond anything that is currently possible in any of the desktop or web based mapping applications.</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;">I propose to implement a system of managing and evaluating properties for generic objects within QGIS. Properties include all settings relating to symbology, such as a line marker&#8217;s width, color, or offset, label settings (eg font size, color, shadow opacity, etc), diagram properties (colors, size, etc) and composer item settings (position, rotation, frame size and color, etc). While currently many of the properties can be set to use &#8220;data defined overrides&#8221;, the properties framework will extend these capabilities by making them both more flexible and easier to use.</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;">This proposal is being driven by a number of factors:</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;">1. To avoid the current multiple duplicate code paths involving storage, retrieval and evaluation of data defined properties and to make it easier to add data defined support to more things (eg diagrams) without incurring even more duplicate code. Currently labeling, symbology and composer all have their own methods for handling data defined properties, which makes maintenance of data defined code very difficult.</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;">2. To allow creation of other property types besides the current &#8220;data defined&#8221; (ie bound to field value or expression result) property, eg time based properties for a future in-built animation framework.</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;">3. To avoid the complexity of requiring users to write their own expressions to map values to colors, sizes, etc and apply scaling functions to these, and instead expose these to users in an interactive, flexible way. Think Mapbox studio&#8217;s approach to zoom level styling (<a href="https://cloud.githubusercontent.com/assets/1829991/17850412/6a0f285e-68a0-11e6-8719-cdf74afd061d.jpg" rel="nofollow">https://cloud.githubusercontent.com/assets/1829991/17850412/6a0f285e-68a0-11e6-8719-cdf74afd061d.jpg</a>), but available for all property types. Eg data defined values can be set to preset ease in/ease out curves, or manually edited curves through an interactive GUI.</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;">4. Enable the possibility of having live project wide colors. Ie a color palette could be created in the project properties, and color based properties &#8220;bound&#8221; to these colors. Altering the color would then automatically update every property which was bound to this preset color. This also brings the possibility of &#8220;color themes&#8221; for maps, eg binding properties to a predefined color types such as &#8220;highlights&#8221;, &#8220;background features&#8221;, etc, and then interactively changing all these color bound properties by applying a color theme to the project.</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;">5. To allow a system of inherited and overridden properties. Eg QGIS default label font overridden by a project default font and finally overridden by label font setting. The proposed composer rewrite (layouts work) would use this property inheritance to bind layout item properties to a dynamic template. Changes in the template would be reflected in all linked layouts, but individual items could overwrite the inherited properties as required. Layout item properties could then be set globally (eg, font size), per project (eg font family), via a &#8220;master template&#8221; and finally individually per layout item.</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;">6. The labelling engine has a need for predefined label styles. Label properties could be set globally, per project, via a predefined style, or overridden for a particular layer.</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;">Technical details regarding this proposal are available in QEP 22 (<a href="https://github.com/qgis/QGIS-Enhancement-Proposals/issues/38" rel="nofollow">https://github.com/qgis/QGIS-Enhancement-Proposals/issues/38</a>). </span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;">I am seeking funding to:</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;">1. Implement the core functionality for the properties framework</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;">2. Port symbology, labeling and diagrams to the framework, and enable data definable control of all appropriate diagram settings (currently diagrams have a very limited data defined control available)</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;">3. Implement the GUI for the property framework, including:</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;">&#8211; a widget for controlling property behaviour</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;">&#8211; interactive widgets for size and color properties (which have been designed to work inside 2.16&#8217;s live layer styling dock)</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;">&#8211; interactive widgets for setting the &#8220;easing&#8221; for properties, with choices of preset ease in/out methods + an interactive curve editor for manual control</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;">If funds are remaining following these items, I will undertake (in order of priority):</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;">4. Bound project colors</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;">5. Begin work on labeling styles</span></p>
<p><b>History:</b><span style="font-weight:400;"> Because I believe so firmly that this framework is required within QGIS, I have been building toward this work through numerous hours of development over the previous 2 years of QGIS releases. There were a number of prerequisite changes required first, such as the implementation of expression contexts. An initial PR (<a href="https://github.com/qgis/QGIS/pull/2857" rel="nofollow">https://github.com/qgis/QGIS/pull/2857</a>) for the properties framework was filed in May 2016, which includes some of the core parts of this proposal. Changes were required based on feedback from that PR , however to date all work on this has been on a volunteer, unsponsored basis and unfortunately I am no longer able to complete such large scale changes as are required by this proposal without funding. Aside from the changes required from the initial PR, significant work remains in implementing GUI, unit tests, and porting symbology and labeling to the new framework.</span></p>
<p><b>Qualifications:</b> <span style="font-weight:400;">I have an extensive history of large-scale contributions to QGIS since 2013 and a proven track record for writing polished UI with extensive unit testing. I&#8217;m passionate about QGIS, being a daily GIS user and strongly believe that this framework is required to take QGIS to the next level of cartographic abilities.</span></p>
<p><b>Implementation Plan:</b> <span style="font-weight:400;">Due to the extensive refactoring and API changes which are required for implementing the properties framework, this work MUST be done in the QGIS 3.0 timeline. If it is not completed during the 3.0 API break period, the amount of work and cost required would substantially increase, and numerous methods across the symbology, labeling and diagrams API would be deprecated. Accordingly this work will be conducted during the QGIS 3.0 timeline, and for greatest testing I would aim to complete the work ASAP (likely complete by late October). Due to the changes required this work would NOT be suitable to backporting to the &gt;= 2.18 branch and will be targeted at QGIS 3.0 only.</span></p>
<p><b>Proposal Link:</b><span style="font-weight:400;">  A QEP detailing technical implemention is available at: <a href="https://github.com/qgis/QGIS-Enhancement-Proposals/issues/38" rel="nofollow">https://github.com/qgis/QGIS-Enhancement-Proposals/issues/38</a>, and an initial PR available at </span><a href="https://github.com/qgis/QGIS/pull/2857"><span style="font-weight:400;">https://github.com/qgis/QGIS/pull/2857</span></a></p>
<p>&nbsp;</p>
<h3><b>Introduce everything necessary for QGIS3 to OSGeo4W (Jürgen Fischer)- €6,000</b></h3>
<p>&nbsp;</p>
<p><b>Details:</b><span style="font-weight:400;"> For QGIS3 we need packages of Qt5, PyQt5 and Python 3 (including many extensions currently available for Python 2).   The goal of this proposal is to introduce all required dependencies to OSGeo4W (32&amp;64bit) that are necessary to build and package QGIS3. The requested amount will cover 60h of work on this.</span></p>
<p><b>History:</b><span style="font-weight:400;"> I also did the packaging of Qt4, PyQt4 and QGIS.  I&#8217;ve also already started to build and package Qt 5.7 using Visual C++ 2015.</span></p>
<p><b>Qualifications:</b> <span style="font-weight:400;">See previous point (or well known history)</span></p>
<p><b>Implementation Plan:</b> <span style="font-weight:400;">I plan on doing it this in Q4 this year to have it available for the release and I don&#8217;t expect significant extra effort to support Windows (ie. if the issues are solved on a platform that already has Qt5 and friends available it should also work on Windows).</span></p>
<h3><b>Implement an inbuilt Task Manager in QGIS for background long running tasks (Nyall Dawson) &#8211; €4,500</b></h3>
<p><b>Details:</b><span style="font-weight:400;"> QGIS requires a centralised, in built task manager to handle background threading of long running analysis tasks. Currently these long running tasks are either conducted while blocking the UI (such as when a snapping index is built for a layer) leading users to conclude that QGIS has frozen, via blocking progress dialogs which prevent interaction with QGIS while the operation proceeds, or via custom threaded implementations. By building a standard framework for handling these long running tasks, we will benefit by:</span></p>
<p><span style="font-weight:400;"><br />
</span><span style="font-weight:400;">1. Avoiding UI blocking tasks, allowing users to continue working while the task is completed.</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;">2. Simplify background task threading for plugin, processing algorithm (and core) developers by exposing a simple API for creating and scheduling long running tasks.</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;">3. Benefit from the stabler code which comes as a result of having a single, well tested implementation of background threading rather than multiple custom implementations of this code.</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;">4. We &#8220;catch up&#8221; to our commercial competitors (ie ArcGIS and MapInfo Professional), who currently have inbuilt background threading of long running tasks already available in their software.</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;">This work was begun in <a href="https://github.com/qgis/QGIS/pull/3004" rel="nofollow">https://github.com/qgis/QGIS/pull/3004</a>, however significant changes are still required before the task manager can be merged into QGIS. It is vital that the task manager implementation is rock solid and with a future proof API which addresses our needs for the 3.x release cycle.</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;">Accordingly, this grant proposal covers:</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;">1. Building off the work started in the pull request, first addressing the feedback received from GitHub and from direct conversations with interested stakeholders and stabilising the API.</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;">2. Completion of the unit tests to cover all parts of the framework.</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;">3. Polish the GUI for interacting with running and completed tasks.</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;">4. Writing documentation for the Python cookbook demonstrating how the task manager should be used from Python code.</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;"><br />
</span><span style="font-weight:400;">(Please note that this proposal does not cover porting any existing code (such as processing) across to the new framework.)</span></p>
<p><b>History:</b><span style="font-weight:400;"> An initial prototype of the work was begun in </span><a href="https://github.com/qgis/QGIS/pull/3004"><span style="font-weight:400;">https://github.com/qgis/QGIS/pull/3004</span></a><span style="font-weight:400;">  </span></p>
<p><b>Qualifications:</b> <span style="font-weight:400;">I have an extensive history of complex changes to QGIS code, and am currently one of the most active QGIS core developers. I have a track record of implementing stable, heavily unit tested code and supporting code I write for extended periods. I am also a daily user of QGIS as a GIS software application, so am invested in making the software as powerful, stable and easy to use as possible!</span></p>
<p><b>Implementation Plan:</b> <span style="font-weight:400;">This work would be completed ASAP to allow for lengthy testing prior to the QGIS 3.0 release, and to allow the maximum time possible for developers to adapt their code and plugins to the new task manager interface.</span></p>
<p><b>Proposal Link:</b><span style="font-weight:400;"> An initial prototype of the work was begun in <a href="https://github.com/qgis/QGIS/pull/3004" rel="nofollow">https://github.com/qgis/QGIS/pull/3004</a>, and a video demonstration is available at </span><a href="https://www.youtube.com/watch?v=7pXBZtWYFJc"><span style="font-weight:400;">https://www.youtube.com/watch?v=7pXBZtWYFJc</span></a><span style="font-weight:400;">   </span></p>
